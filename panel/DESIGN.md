# Dashboard And Sidebar Refresh Design

## 背景

本次改造目标是参考提供的仪表盘截图，重写 `dashboard` 首页与 `sidebar` 视觉样式，同时保留现有菜单结构、路由结构和功能边界，不做信息架构重排。

项目实际由两个独立仓库协同完成：

- `codeProxy`：管理端前端页面、布局、交互与文案。
- `CliRelay`：管理端首页需要的新聚合指标与趋势接口。

## 目标

- 将首页改造成更接近截图的运营面板风格，突出 4 个顶部 KPI、系统状态区和底部趋势区。
- 重写 sidebar 视觉，包括品牌区、导航项高亮、底部账号卡片和升级卡片的布局语言。
- 尽量复用现有 `SystemMonitorSection` 中已经具备的数据，避免重复后端工作。
- 为首页补齐趋势数据接口，使顶部 KPI 卡片和底部折线图不再只有静态数字。

## 非目标

- 不调整现有菜单顺序、路由层级和页面入口。
- 不新建截图中不存在的业务模块页面。
- 不引入新的前端状态管理方案。
- 不做与首页/侧边栏无关的大规模重构。

## 现状结论

### 前端

- `src/modules/dashboard/DashboardPage.tsx` 当前只有 4 张基础 KPI 卡片和一个 `SystemMonitorSection`。
- `src/modules/ui/AppShell.tsx` 已经承担 sidebar、header、主题切换、语言切换与登出逻辑。
- `SystemMonitorSection` 已经具备健康评分、运行时间、Go 协程、数据库、日志存储、网络流量、渠道平均延迟、RPM/TPM、磁盘可用等大部分截图中部/底部指标。

### 后端

- `GET /dashboard-summary?days=N` 当前只返回聚合 KPI 数字和一些账户数量统计。
- `GET /system-stats` / `GET /system-stats/ws` 已返回系统资源、流量、吞吐和渠道延迟，但没有首页趋势图需要的时间序列。
- SQLite `request_logs` 表可以支撑按天或按时间桶聚合请求数、成功率、Token 和吞吐趋势。

## 设计方案

### 1. Sidebar 视觉重写

保留现有 `NAV_ITEMS` 和跳转行为，只改视觉呈现：

- 顶部品牌区改成更明显的产品标识和标题排版。
- 导航项改成更厚实的卡片式高亮，当前页使用蓝色发光/渐变风格。
- 整体背景改为浅色柔和渐变，增加边框、模糊和阴影层次。
- 在侧边栏底部加入：
  - 一个升级提示卡片，作为纯视觉模块，不绑定新业务。
  - 当前账号信息卡片，复用现有登出上下文，不新增账户系统依赖。

### 2. Dashboard 布局重组

首页结构调整为三段：

- 顶部：标题、副标题、时间范围切换、刷新按钮。
- 中上：4 张 KPI 卡片，每张卡片带图标、主指标、说明文字和微型趋势线。
- 中下到底部：系统状态大卡片，内部继续细分为健康评分、基础资源卡、吞吐趋势图、日志与磁盘摘要等。

其中：

- 顶部 4 个 KPI 保留现有语义：
  - 总请求数
  - 成功率
  - 总 Token 数
  - 失败请求
- `SystemMonitorSection` 延续现有数据能力，但重新排版为更贴近截图的网格布局。
- 新增底部趋势图，显示近 50 个采样点内的 RPM 与 TPM 走势。

### 3. 后端接口扩展

扩展 `dashboard-summary` 返回结构，在保持现有字段兼容的前提下新增：

- `trends.request_volume`
  - 用于顶部总请求数卡片微趋势图。
- `trends.success_rate`
  - 用于成功率卡片微趋势图。
- `trends.total_tokens`
  - 用于 Token 卡片微趋势图。
- `trends.failed_requests`
  - 用于失败请求卡片微趋势图。
- `throughput_series`
  - 用于首页底部 RPM/TPM 双折线图。
- `meta.generated_at`
  - 用于展示更新时间，减少前端自行猜测。

聚合口径：

- `days=1`：按小时聚合。
- `days=7` / `days=30`：按天聚合。
- `throughput_series`：取最近窗口内等粒度样本，统一返回 `label`、`rpm`、`tpm`。

### 4. 测试策略

前端：

- 为 `DashboardPage` 增加结构测试，确保新布局包含趋势区与新的 KPI 表达。
- 为 `AppShell` 增加 sidebar 视觉结构测试，确保品牌区、升级卡片和账号卡片存在。

后端：

- 为 `dashboard-summary` 增加接口测试，验证新增趋势字段结构和基础统计值。
- 为趋势聚合函数增加单元测试，验证按天/按小时分桶结果。

## 风险与处理

- 风险：两个仓库独立提交，前端可能先依赖后端新字段。
  - 处理：前端对新增字段做向后兼容兜底；后端先补接口测试和实现。
- 风险：首页视觉改动较大，容易影响现有响应式布局。
  - 处理：保持 `AppShell` 现有移动端开关逻辑，仅重写样式结构，不改导航状态机。
- 风险：趋势图需要新数据结构，若后端数据为空会导致空白或报错。
  - 处理：前端统一规范空数组渲染，展示占位状态。
